Health Check (Specification)

This document describes the health check specification that is to be implemented/provided by all ipipeline resources [e.g. products, components, endpoints, etc.]

This specification follows the upstream specification described in the api-healthcheck internet draft work in progress [1] [2], and extends that specification.

Application

While the upstream specification is specific to apis, this specification is applicable to all http[s] services, not just apis

Url

The path [3] component of a health url must be the literal string "/health", and must be located at the base of the service/api url

Examples

Headers

GET /health HTTP/1.1 Host: component.iplsys.net Accept: application/health+json

HTTP/1.1 200 OK
Content-Type: application/health+json
Cache-Control: max-age=3600
ipipeline-health-status: pass
Connection: close

Json

{
   "status":"pass",
   "serviceId":"f03e522f-1f44-4062-9b55-9587f91c9c41",
   "description":"health of component api",
   "version":"1",
   "releaseId":"1.2.2",
   "checks":{
      "database:responseTime":[
         {
            "componentId":"dfd6cf2b-1b6e-4412-a0b8-f6f7797a60d2",
            "componentType":"component",
            "observedValue":250,
            "observedUnit":"ms",
            "status":"pass",
            "affectedEndpoints":[
               "/users/{userId}",
               "/customers/{customerId}/status"
            ],
            "time":"2021-01-17T03:36:48Z",
            "output":""
         }
      ],
      "esb_whatever:responseTime":[
         {
            "componentId":"dfd6cf2b-1b6e-4412-a0b8-f6f7797a60d2",
            "componentType":"datastore",
            "observedValue":75,
            "status":"warn",
            "time":"2021-01-17T03:36:48Z",
            "output":"",
            "links":{
               "self":"http://component.iplsys.net/dbnode/dfd6cf2b/health"
            }
         }
      ],
      "monitor":[
         {
            "interval":{
               "normal":30,
               "failure":15,
               "recovery":30
            },
            "failuresbeforeaction":2,
            "successesbeforeaction":11
         }
      ]
   }
}

base url: https://component.iplsys.net/service
health url: https://component.iplsys.net/service/health

base url: https://component.iplsys.net/api
health url: https://component.iplsys.net/api/health

Http messages

since there may be clients unable to parse a json message, to provide for better interoperability, the http message for a health check must include the following:

status codes [1]

  • for a status field value of either "pass" or "warn", the https status code should be 200 [ok]

  • for a status field value of "fail", the http status code should be 503 [service unavailable]

deviation from this is allowable, but discouraged, and must not deviate from the upstream specification. any deviation must be accompanied by supporting documentation
use of the 5xx status code class is a quirk of the health check specification, as it does not prohibit returning content, but may be interpreted to conflict with the http response status codes specification [4]

response headers [2]

the http response must contain an ipipeline-health-status header, providing the same value as is reported by the mandatory root "status" field in the json response.

example:

ipipeline-health-status: pass

response format

in addition to the specification in the api-healthcheck internet draft, the following apply:

  • in addition to the mandatory root field ["status"] declared in the internet draft, the following fields [as described in the upstream specification] are mandatory for all ipipeline implementations:

    • serviceid

    • description

    • version

    • releaseid

  • serviceid must be a version 4 uuid which conforms to the specification described in rfc 4122 [5], and must be registered in the component uuid registry [6]

  • where an implementation includes a checks object, the following fields [as described in the upstream specification] are mandatory for each subcomponent:

    • componentname

    • measurementname

    • status

    • componentid

    • componenttype

    • observedvalue

    • observedunit

    • time

monitor object

the monitor object contains an array which declares the intervals and frequencies with which the given health state should be queried. each implementation’s response format should include a monitor object, with the following mandatory content:

interval [object]

the interval object must have the following keys:

  • normal: the number of seconds between checks during normal conditions

  • failure : the number of seconds between checks during failure conditions

  • recovery : the number of seconds between checks during recovery conditions
failuresbeforeaction [unsigned integer]

in a failure state, the number of failures that should be occur before an action is taken
successesbeforeaction [unsigned integer]

in a recovery state, the number of successes that should be occur before an action is taken

if a given implementation does not publish a monitoring object, the following default values are assumed:

  • interval

    • normal condition - 30

    • failure condition - 15

    • recovery condition - 30

  • failures before action - 2

  • successes before action - 11

security

inherently, health check resources often contain confidential information, and as such, must implement appropriate security controls as per ipipeline policies

examples can be found in the /examples/ directory
