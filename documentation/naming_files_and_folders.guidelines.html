Naming Files and Folders (Guidelines)

Table of Contents
  • General Guidelines
  • Separate words in file names with underscores
  • Symbols and filenames
  • Use a flat folder structure
    • Exceptions for consistency

Naming conventions are hugely important to maintainability and readability. This guide recommends naming conventions for the file name and the symbol name.

General Guidelines

✅ Do follow a pattern that describes the symbol’s feature then its type. The recommended pattern is feature.type.adoc ✅ Do make file and directory names lowercase ✅ Do name files and files to clearly convey their intent ✅ Do use only standard ASCII alphanumeric characters in file and directory names.

❌ Don’t use generic page names such as document1.html.

Why? Naming conventions help provide a consistent way to find content at a glance. Consistency within the project is vital. Consistency with a team is important. Consistency across a company provides tremendous efficiency.

Why? Nnaming conventions help find desired code faster and make it easier to understand.

Why? Names of folders and files should clearly convey their intent. For example, templates/architecture_decision_record.template.adoc` should contain a template for architecture decision records.

Separate words in file names with underscores

✅ Do use underscores, not dashes (hyphens) to separate words, for example, design_guide.template.adoc.

Why? Grammatically, hyphens are for hyphenating words. using them for other things introduces potential ambiguity. Also, visually, underscores more closely represent the whitespace they’re replacing.

Symbols and filenames

✅ Do use consistent names for all assets named after what they represent. ✅ Do give the filename the conventional suffix representing it’s symbol (such as .template.adoc, .adr.adoc, .guidelines.adoc, .standard.adoc) for a file of that type.

Why? Consistent conventions make it easy to quickly identify and reference assets of different types.

Use a flat folder structure

We have chosen to adopt as flat a folder structure as possible to make things easier to find.

Make folder names lowercase. Use underscores, not hyphens, to separate words, for example, health_check. Use only standard ASCII alphanumeric characters in file and directory names.

Exceptions for consistency

If you’re adding to a directory where everything else already uses underscores, and it’s not feasible to change everything to underscores, it’s okay to use hyphens to stay consistent.

For example, if the directory already has lesson_1.jd, lesson_2.jd, and lesson_3.jd, it’s okay to add your new file as lesson_4.jd instead of lesson-4.jd. However, in all other situations, use hyphens.

✅ Recommended: avoiding_cliches.jd ✅ Sometimes OK: avoiding-cliches.jd

❌ Not recommended: avoidingcliches.jd ❌ Not recommended: avoidingCliches.jd ❌ Not recommended: avoiding_clichés.jd

Other exceptions It’s okay to have some inconsistency in filenames if it can’t otherwise be avoided. For example, sometimes tools that generate reference documentation produce filenames based on different style requirements or based on the design and naming conventions of the product or API itself. In those cases, it’s okay to make exceptions for those files.
