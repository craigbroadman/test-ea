This document applies to architecture designs that are created using tools such as draw.io, lucidchart and visio.

Style Guide

structure/organization

  • documentation is organized by component, not type. e.g. all documentation for given component is stored together, regardless of if it is graphical [e.g. diagrams], textual [e.g. written specifications], etc.

syntax/style

  • alignment

    • all content should be aligned

indentation

  • indentation must be equal to 4 spaces, tabs or spaces are ok but must be consistent throughout the document/project

font

  • size

    • font size must be consistent throughout a given document

  • family

    • font family must be consistent throughout a given document

  • color

    • freeform text must be black

    • shape text may use its natural color

character case

  • all text should be lower case

labeling

  • all shapes must be labeled

  • labels should avoid interferences with other shapes [including lines]

lines

  • width should be 1 pixel

  • orthogonal lines should use rounded corners

  • all lines in a document must use the same style

  • lines should intersect other lines as little as possible

  • lines should have the minimum number of turns neccesary

  • line descriptions should express the application specific functional purpose [not layer 3 protocol/port]

  • lines should include arrows which express direction of communication

  • line colors should be documented in the legend, and include layer 3 protocol and port

  • layer 3 protocol/port color coding should follow the template

  • layer 3 protocols and ports that aren’t on the template should may use any color, but should avoid existing template colors where possible

shapes

  • dimensions

    • a shape has a dominant dimension, which is the smaller of its width or height

    • a shape’s dominant dimension should be 30 pixels

    • all shapes in a document should have the same dominant dimension

    • all shapes should retain their porportions

  • all shapes should be placed on the appropriate layer [e.g. content, not background, etc.]

  • a shape’s text should be positioned so it interferes with other diagram elements [e.g. lines, etc] as little as possible

  • agnostic/generic shapes should be used where possible

  • provider/services specific shapes may be used where appropriate or necessary

  • alignment

    • shapes should share alignment where possible

legend

terminology

jargon

don’t use nonsense words for objects, software, processes, etc. if a word is just an obscure substitute for some other well known word that already existed, just use that word.

acronyms

example: to refer to a computer, rather than using terms like box, machine, etc., use the word computer

manufactured acronyms should not be used. constructing ad-hoc acronyms in place of existing text or natural language/speech should be avoided. existing, well known, acronyms are acceptable.

abbreviations

example: tcp is a well known, long established acronym that is widely accepted to mean transport control protocol, and as such, as acceptable to use. pov, used in place of point of view, is unnecessary to communicate effectively, and can require an explanation that inhibits communication.

abbreviations should not be used. using the shortened form of a word or phrase contributes to potential inconsistency and lack of clarity.

shorthand

example: use "management", not mgmt or mgt.

shorthand notation should not be used. similar in nature to acronyms and abbreviations, use of symbols or other indirect references in place of common language or writing adds to the potential for confusion and does not promote clear, effective communication.

Maintainance

summary

this document describes the document maintenance standards defined for architecural design documents, etc.

maintenance of documents

current

  • component documentation will include infrastructure profile data

  • point-in-time infrastructure profile data will be compared to repository contents

  • as design changes occur infrastructure profile data will be updated

future/proposed

  • automation of current

  • additional granularity where identified

design update request

  • create design change request issue

    • https://github.com/ipipeline/infrastructure-architecture/issues

notes

Source Control standards

general

all documentation must be in a format that is conducive to version control.

  • design documentation must be in xml format, and must use the jgraph specification

    • https://github.com/jgraph

  • reference documentation must use the asciidoc format, and must use the .adoc filename suffix

    • https://asciidoc.org/

repository structure

  • documentation should reside within the product’s repository, in a directory named "documentation"

notes

  • structure

    • option 1 - create repos according to product [laserapp-documentation, laserapp-infrastructure, laserapp-application]

    • option 2 - store in a docs` folder within the repo containing your code

  • branches

    • jira summary should match branch description

    • merge commits should use jira summary as commit summary

    • merge commits should use jira description as commit description

    • naming convention

      • component[product, etc]/category/task

      • categories

        • new

        • feature

        • bugfix

        • junk

  • access

    • repository/document management access controls

      • read - everyone else, as needed

      • write - minimal, limited to architects
