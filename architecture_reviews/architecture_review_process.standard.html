Overview

This documentation describes the Architecture Review process and is intended as a guide for the Solution Architects who may be called upon to conduct these reviews, and the project teams who request them

If a code review is looking at a finished piece of code to ensure in conforms with the specification and follows the appropriate coding standards, then an architecture review is looking at a proposed design to assess if it conforms with iPipeline patterns, and that there is not already code in existence that could be adopted/adapted.

The Architecture Review Process

  1. Request

    • Project teams should initiate the review process by submitting a request using the Architecture Review Form

    • Emails will be sent automatically to the relevant solution architects, and a review card created in their team channel where they can log their findings

  2. Discovery

    • If necessary, the solution architects should liaise with the team requesting the review to understand the proposed solution

  3. Review

    • The solution architects should review the proposal and complete the review card indicating the number of low, medium, high, and critical findings, and attaching any relevant documentation

  4. Publish Findings

    • On completion of the review card, the project team will be notified by email

  5. Action

    • The project team should review the findings and take the appropriate action(s)

When should the team request a review

Unfortunately it is not possible to provide an exhaustive set of circumstances in a which a development team should seek an architecture review. As a guide however, if the change is a defect fix or a small change to some business logic then an architecture review is probably not required. If it is a new service, a new technology, a new design pattern, a new integration, a significant refactoring, or a complex piece of business logic (especially if it crosses multiple data domains), then an architecture review probably is required.

Findings

An architecture review should be considering whether a proposed design is in line with iPipeline patterns (as defined by the architecture community as a whole) and whether there is already an existing solution that could be reused.

In addition, solution architects should also consider any product specific standards, over and above those specified by the community, and how well these have been followed.

Feedback should be left in the way of findings, classified as low, medium, high or critical depending on their severity. A review can have multiple findings. Although some discretion will be needed when determining a findings severity, you can use the below examples as a guide.

Table 1. Classification of Findings
Category Definitions

Low

  • EA-L01 - The proposed solution uses a different version of a technology than in the documented best practice

  • EA-L02 - The proposed solution uses a technology or design pattern for which there is no standard (note this is a call to action for the architecture community to address the missing standard)

Medium

  • EA-M01 - The proposed solution uses a design pattern which deviates from the documented best practice in a minor way

  • EA-M02 - The proposed solution uses a technology or design pattern in a way that deviates from existing use cases

  • EA-M03 - The proposed solution uses a technology or design pattern which deviates from a general interim practice

High

  • EA-H01 - The proposed solution uses a design pattern which deviates from the documented best practice in a material way

  • EA-H02 - The proposed solution uses a new technology or design pattern which should be discussed with the architecture community before being put into use

  • EA-H03 - The proposed solution could have relevance to other iPipeline teams but is not designed to be reusable

  • EA-H04 - The proposed solution incurs questionable cost

  • EA-H05 - The proposed solution would benefit by maturing an existing process

Critical

  • EA-C01 - There is already an existing solution that could be reused/adapted to meet the requirement

  • EA-C02 - The proposed solution uses a technology or design pattern which is wholly different to the documented best practice

  • EA-C03 - The proposed solution creates a critical security vulnerability

  • EA-C04 - The proposed solution incurs sizable questionable cost

Escalations

If an architecture review assigned to you is outside your area of expertise, or you are unsure if the proposed design aligns with best practice, you have the option to escalate the review to the enterprise architecture team.

Remember that for questions on best practice you can always reach out informally to either the enterprise architecture team or the architecture community via email, chat, or the Ask the Architects forum! Please try these channels first before escalating a review
